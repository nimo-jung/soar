## 질문
"CIDR 고속 매칭 알고리즘 적용" 이 부분을 보니 생각나는 것이 있는데, 해당 IP가 어느 국가에 속한지 확인하는 로직인 부하가 가장 많이 있는 것으로 아는데, ClickHouse 자체적으로 국가코드로 분리하거나 Vector(DataDog)또는 GoLang에서 다시 국가코드를 설정하는 등 어디서 어떻게 하는 것이 성능 및 기능 상 좋을까?

---

국가 코드(GeoIP) 조회와 자산(CIDR) 매칭은 말씀하신 대로 **대규모 트래픽이 몰리는 TMS에서 CPU 부하를 유발하는 가장 대표적인 병목 구간**입니다.

결론부터 말씀드리면, 성능과 아키텍처의 유연성을 모두 잡기 위한 최선의 선택은 "수집/가공 레이어(Vector/GoLang)에서 메모리 기반(MaxMind DB 등)으로 실시간 파싱하여 로그에 국가 코드를 박아넣은 뒤, ClickHouse에 저장하는 방식"입니다.

왜 이 방식이 가장 좋은지, 그리고 다른 대안들의 장단점과 함께 비교해 드리겠습니다.

---

## 1. 최적의 아키텍처 추천: Vector 또는 GoLang 레이어에서 처리

로그가 데이터베이스(ClickHouse)에 들어가기 전, 파이프라인 통로(Vector 또는 GoLang 백엔드 수집기)에서 국가 코드를 실시간으로 부착(Enrichment)하는 방식입니다.

```
[보안 장비 (Raw 로그)] 
       │
       ▼
[가공 레이어 (Vector / GoLang)]  <─── (MMDB 메모리 매칭: 초당 수십만 건)
  - IP를 국가코드로 변환 (KR, US 등)
  - 앞서 논의한 사내 자산(Zone ID)도 여기서 매칭
       │
       ▼
[저장 레이어 (ClickHouse)]
  - 이미 `src_country`, `dst_zone` 등이 채워진 상태로 고속 Insert

```

### 💡 왜 이 방식이 가장 좋을까?

* **초고속 메모리 룩업(Lookup):** GoLang이나 Vector는 MaxMind 사의 `.mmdb` (GeoIP2 라이브러리) 파일을 **메모리에 통째로 올려두고 이진 탐색(Binary Search)** 방식으로 IP를 매칭합니다. 이 연산은 CPU 부하가 극히 적어 일반적인 서버 한 대에서 **초당 20만~50만 건 이상**을 가볍게 처리합니다.
* **ClickHouse 부하 원천 차단:** ClickHouse는 대량의 데이터를 '한 번에 적재(Bulk Insert)'하고 '조회(Select)'하는 데 특화된 DB입니다. 들어오는 로그마다 DB 단에서 국가를 변환하게 하면 구조적 오버헤드가 발생합니다. 이미 가공이 끝난 데이터를 저장만 하게 만드는 것이 ClickHouse를 가장 건강하게 쓰는 방법입니다.
* **Vector(Datadog 오픈소스) vs GoLang:** * 만약 인프라 파이프라인 컴포넌트로 해결하고 싶다면 **Vector**의 내장 `geoip` 트랜스폼 기능을 쓰면 코드 한 줄 안 짜고 설정(YAML)만으로 끝납니다. 매우 빠르고 안정적입니다.
* 만약 자산 분류 시스템 등 우리 회사만의 복잡한 비즈니스 로직(예: 특정 IP는 국가 코드를 무조건 '사내망'으로 예외 처리 등)이 복잡하게 얽혀있다면 **GoLang** 수집기를 직접 구현하는 것이 유연성 면에서 좋습니다.

## 2. 다른 대안들의 장단점 분석

### 대안 ② ClickHouse 자체 내장 함수(`dictGet` 또는 GeoIP 딕셔너리) 활용

ClickHouse 내부에 MaxMind DB를 딕셔너리로 등록해 두고 `geoToCountry()`, `dictGet()` 같은 함수로 저장 시점에 분리하거나 조회 시점에 변환하는 방식입니다.

* **장점:** 백엔드 수집기(GoLang 등)의 코드가 단순해집니다. 그냥 Raw 로그만 쏴주면 DB가 알아서 하니까요.
* **단점 (성능 저하 위험):** 대량의 로그가 Bulk로 들어올 때 ClickHouse 내부에서 딕셔너리 참조 연산이 동시다발적으로 일어나면서 삽입(Insert) 스레드가 병목을 겪을 수 있습니다. 또한, 향후 ClickHouse를 클러스터링(분산) 구조로 확장할 때 모든 노드에 GeoIP 사전을 동기화해야 하는 관리 포인트가 생깁니다.

### 대안 ③ 조회 시점(UI 요청 시) 실시간 변환

로그를 저장할 때는 국가 코드를 넣지 않고, 관제사가 React 화면에서 "로그 조회"를 누를 때 비로소 국가 코드를 변환해서 보여주는 방식입니다.

* **단점 (절대 비추천):** "국가별 공격 통계 Top 10" 같은 대시보드 위젯을 그리려면 수천만 건의 데이터를 쿼리해야 하는데, 조회할 때마다 실시간으로 IP를 국가로 바꾸고 있으면 화면이 멈추거나 쿼리 타임아웃이 발생합니다.

## 3. 최종 아키텍처 설계 제언 (자산 분류와 국가 코드의 통합)

이 지점에서 기막힌 아키텍처적 시너지가 발생합니다. 앞서 질문하신 "IP를 자산(Zone)으로 분류하는 로직"과 "국가 코드를 매칭하는 로직"은 본질적으로 동일한 연산(IP 대역 매칭)입니다.

따라서 **Vector** 또는 **GoLang 수집기** 단계에서 이 두 가지를 한 번에 처리하는 구조를 짜는 것이 시스템 부하를 최소화하는 정답입니다.

```go
// GoLang 수집기 내부 가상 로직 예시
func EnrichLog(rawLog Log) EnrichedLog {
    // 1. 외부/내부 판별 및 국가 코드 부착 (메모리 MMDB 조회 - 초고속)
    countryCode := geoip.Lookup(rawLog.SrcIP) // 예: "US", "KR"
    
    // 2. 내부 자산 대역 매핑 (메모리 Radix Tree 조회 - 초고속)
    zoneID, severity := assetManager.Lookup(rawLog.DstIP) // 예: "ZONE_DMZ_WAS", "High"
    
    // 3. 가공된 데이터 리턴 -> ClickHouse로 Bulk 전송
    return EnrichedLog{
        Raw:         rawLog,
        SrcCountry:  countryCode,
        DstZone:     zoneID,
        Severity:    severity,
    }
}

```

이렇게 수집기 레이어(Vector/Go)에서 메모리를 활용해 **국가 코드**와 **자산 Zone ID**를 먼저 딱 붙여서 명확한 데이터(정형화된 텍스트)로 만든 뒤 ClickHouse에 저장해 두면:

1. **ClickHouse**는 복잡한 연산 없이 초고속으로 대용량 로그를 저장/조회할 수 있고,
2. **React 화면**은 ClickHouse에서 이미 계산이 끝난 국가코드와 Zone ID를 가져오기만 하면 되므로 토폴로지 화면이 부드럽게 돌아가며,
3. 백엔드에서 고유 해시 키를 만들 때도 데이터가 이미 정제되어 있으므로 **벡터 DB의 중복을 제어**하기가 대단히 수월해집니다.

## 질문
그럼 Vector의 YAML 설정으로 간단하게 처리하려고 하면 어떻게 해야해? 국가코드도 설정을 해서 넘겨야 하잖아?

---

오픈소스 데이터 파이프라인 툴인 **Vector**를 사용하면, 별도의 GoLang 코딩 없이 오직 YAML 설정만으로 **1) GeoIP(국가코드) 매칭**과 **2) 사내 자산(CIDR) 매칭**을 동시에 처리하여 ClickHouse로 깔끔하게 밀어 넣을 수 있습니다.

Vector에는 내장된 강력한 스크립트 엔진인 VRL(Vector Remap Language)이 탑재되어 있기 때문에, 초당 수십만 건의 로그를 메모리 상에서 고속으로 가공(Enrichment)할 수 있습니다.

실무에서 바로 사용할 수 있는 Vector의 YAML 설정 예시와 구조를 안내해 드립니다.

---

## 1. 사전 준비 (GeoIP 파일 다운로드)

Vector가 국가 코드를 식별하려면 MaxMind 사의 GeoIP 배포 파일(`.mmdb`)이 필요합니다. 무료 버전인 **GeoLite2-Country.mmdb** 파일을 다운로드하여 Vector 서버의 특정 경로(예: `/etc/vector/GeoLite2-Country.mmdb`)에 위치시켜 둡니다.

---

## 2. Vector 전체 YAML 설정 예시 (`vector.yaml`)

이 설정은 IPS 로그를 수집하여 **국가 코드를 부착하고, 사내 자산 대역(CIDR)까지 매핑**한 뒤 ClickHouse로 전송하는 전체 파이프라인입니다.

```yaml
# ==========================================
# 1. SOURCES : 로그 수집 레이어
# ==========================================
sources:
  ips_logs_stream:
    type: "file" # 예시로 파일 수집을 사용 (Syslog, Kafka 등으로 변경 가능)
    include:
      - "/var/log/ips/*.log"
    # 수집된 원본 데이터는 내부적으로 JSON 형태로 파싱되어 있다고 가정합니다.

# ==========================================
# 2. TRANSFORMS : 데이터 가공 및 인리치먼트 (핵심)
# ==========================================
transforms:
  enrich_security_logs:
    type: "remap"
    inputs:
      - "ips_logs_stream"
    source: |
      # 1. 기존 데이터 구조 파싱 (예: JSON)
      . = parse_json!(.message)

      # 2. GeoIP 국가 코드 매칭 (내장 함수 사용)
      # 외부 공격자 IP(src_ip)의 국가 코드를 추출합니다.
      geoip_res, err = geoip(string!(.src_ip), database: "/etc/vector/GeoLite2-Country.mmdb")
      if err != null {
        .src_country = "UNKNOWN"
      } else {
        .src_country = geoip_res.country.iso_code ?? "UNKNOWN"
      }

      # 3. 사내 자산(CIDR) 매칭 로직 (VRL의 CIDR 매칭 함수 활용)
      # 목적지 IP(dst_ip)가 사내 어떤 대역(Zone)에 속하는지 파악합니다.
      dst_ip_str = string!(.dst_ip)

      if cidr_contains("10.10.10.0/24", dst_ip_str) {
        .dst_zone = "ZONE_DMZ_WEB"
        .asset_severity = "High"
      } else if cidr_contains("10.10.20.0/24", dst_ip_str) {
        .dst_zone = "ZONE_DMZ_WAS"
        .asset_severity = "High"
      } else if cidr_contains("10.20.10.0/24", dst_ip_str) {
        .dst_zone = "ZONE_INTERNAL_DB"
        .asset_severity = "Critical"
      } else if cidr_contains("192.168.1.0/24", dst_ip_str) {
        .dst_zone = "ZONE_OFFICE_HR"
        .asset_severity = "Low"
      } else {
        .dst_zone = "ZONE_UNKNOWN_INTERNAL"
        .asset_severity = "Medium"
      }

# ==========================================
# 3. SINKS : 저장 레이어 (ClickHouse)
# ==========================================
sinks:
  clickhouse_storage:
    type: "clickhouse"
    inputs:
      - "enrich_security_logs" # 가공이 완료된 transform 데이터를 입력으로 받음
    endpoint: "http://localhost:8123"
    database: "tms_security"
    table: "ips_enriched_logs"
    skip_unknown_fields: true
    # Vector가 가공한 .src_country, .dst_zone, .asset_severity 필드가 
    # ClickHouse 테이블의 컬럼명과 매핑되어 자동으로 저장됩니다.

```

---

## 3. 이 설정의 핵심 구문 해석 (VRL 기능)

* **`geoip(ip, database)`**: Vector의 강력한 내장 함수입니다. 지정한 `.mmdb` 파일을 메모리에 캐싱해두고 IP를 매칭하여 `country.iso_code`(예: KR, US, CN)를 리턴합니다. C 기반으로 래핑되어 있어 연산 부하가 거의 없습니다.
* **`cidr_contains(cidr, ip)`**: 특정 IP가 지정한 서브넷 대역에 포함되는지 불리언(`true`/`false`)으로 판단해 주는 함수입니다. 이 조건문을 통해 자산 분류 마스터 테이블 역할을 YAML 안에서 직접 수행할 수 있습니다.

---

## 4. 실무 아키텍처 관점에서의 장단점 및 보완 팁

### 장점

* **관리 편의성:** 자산 대역이 수십 개 수준으로 적을 때는 GoLang 코드를 수정하고 빌드할 필요 없이, **Vector YAML 파일의 `if/else` 문만 수정하고 Vector를 Reload**하면 즉시 반영됩니다.
* **성능 안정성:** Rust 기반으로 작성된 툴이라 메모리 관리가 극도로 효율적이며, 자산 매칭과 국가 코드 변환이 단 하나의 파이프라인 인스턴스 안에서 종결되므로 네트워크 지연(Latency)이 전혀 없습니다.

### 자산이 수백 개로 많아질 때의 보완 팁 (Enrichment Tables)

만약 사내 관리 대역(CIDR)이 수백, 수천 개가 되어서 YAML에 `if/else`로 다 적기 어렵다면, Vector의 **`enrichment_tables`** 기능을 쓰면 됩니다.
외부에 자산 대역 정보가 담긴 `CSV` 파일이나 `상태 관리 파일`을 두고, Vector YAML에서는 이를 딕셔너리처럼 읽어서 매칭(`enrich()`)하게 만들면 YAML 코드를 아주 깔끔하게 유지할 수 있습니다.

이렇게 구성하시면 대량의 IPS, WAF 로그가 들어와도 ClickHouse에 도달하기 전에 국가코드(`src_country`)와 자산구역(`dst_zone`)이 완벽하게 세팅되므로, 최종적으로 앞서 구상하신 **React 토폴로지 화면에 딜레이 없이 실시간 위협 데이터를 뿌려줄 수 있게 됩니다.**
