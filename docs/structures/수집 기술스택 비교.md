SOAR 시스템의 핵심인 **데이터 수집 레이어** 구축을 위해 고민 중이신 4가지 기술 스택(솔루션 vs 자체 제작)을 수집 성능, 개발 속도, 유지보수 관점에서 명확하게 비교해 드리겠습니다.

## 돋보기 1. Vector (Datadog) — *추천안*

Rust로 작성된 차세대 고성능 로그 수집기입니다.

* **수집 성능**: **최상.** 가비지 컬렉션(GC)이 없어 초당 수십만 건의 로그가 유입되어도 지연(Latency Spike)이 없고 CPU/메모리 소모가 극도로 적습니다.
* **개발 속도**: **가장 빠름.** 이미 검증된 `syslog`, `http_client/server` 등의 소스가 내장되어 있어 `.toml` 설정 파일 정의와 VRL 스크립트 작성만으로 며칠 만에 수집 파이프라인을 구축할 수 있습니다.
* **유지보수 관점**: **최상.** 시스템 구동 중에 설정이 바뀌어도 무중단(Hot Reload)으로 반영됩니다. 새로운 벤더사 장비가 추가될 때 코드를 다시 컴파일하거나 서버를 끌 필요가 없어 SOAR 관리 도구와 연동하기 가장 좋습니다.

## 돋보기 2. 자체 제작 (C++)

모든 네트워크 소켓 통신과 파싱 로직을 C++ 표준 라이브러리로 직접 구현하는 방식입니다.

* **수집 성능**: **최상.** 하드웨어 자원을 100% 통제할 수 있으므로 이론상 가장 빠르고 가벼운 전용 수집기를 만들 수 있습니다.
* **개발 속도**: **가장 느림.** Syslog RFC 표준 구현, SNMP 멀티스레딩 풀링, REST API 클라이언트 및 예외 처리, 메모리 관리(포인터, 누수 방지) 등을 바닥부터 직접 짜야 하므로 프로토타입 제작에만 수개월이 걸립니다.
* **유지보수 관점**: **매우 어려움.** 벤더사별로 상이한 로그 파싱 규칙이 추가될 때마다 C++ 코드를 수정하고, 빌드하고, 테스트한 뒤 바이너리를 다시 배포해야 합니다. 개발 인력 유실 시 유지보수 지옥이 될 위험이 큽니다.

---

## 돋보기 3. Logstash (Elastic)

ELK 스택의 전통적인 자바(JVM) 기반 로그 수집기입니다.

* **수집 성능**: **낮음~보통.** 대량의 로그 전처리 시 JVM 가비지 컬렉션으로 인해 멈춤 현상(Stop-the-world)이 발생할 수 있으며, 데이터 처리를 위해 수 GB 이상의 RAM을 기본적으로 먹는 헤비한 솔루션입니다.
* **개발 속도**: **빠름.** 수많은 인풋/아웃풋 플러그인과 레퍼런스가 시장에 깔려 있어 진입 장벽이 낮습니다.
* **유지보수 관점**: **보통.** 설정 파일 구조가 직관적이지만, 대규모 트래픽 튜닝 시 JVM 메모리(Heap Size) 관리와 성능 최적화 까다롭고 솔루션 자체가 무거워 컨테이너 기반 확장 시 인프라 비용이 많이 듭니다.

## 돋보기 4. 자체 제작 (GoLang 또는 Python)

GoLang이나 Python의 내장 네트워크 라이브러리를 활용해 수집 엔진을 직접 개발하는 방식입니다.

* **수집 성능**:
* **GoLang**: **우수함.** 고루틴(Goroutine) 덕분에 수천 개의 멀티 테넌트 소켓을 띄워도 가볍고 네이티브에 준하는 속도가 나옵니다.
* **Python**: **낮음.** GIL(글로벌 인터프리터 락) 제약으로 인해 초당 수만 건의 Syslog 스트리밍 연산을 소화하기 어렵고 병목이 심합니다.


* **개발 속도**: **보통.** C++보다는 오픈소스 라이브러리가 많아 개발이 빠르지만, 여전히 에러 핸들링과 프로토콜 수신 스케줄러를 직접 코딩해야 합니다.
* **유지보수 관점**: **보통.** 파싱 비즈니스 로직과 네트워크 I/O 로직이 엔진 코드 내부에 뒤섞이기 쉽습니다. 앞서 언급한 'ClickHouse로 파싱 오프로딩' 전략을 취하지 않는다면, 벤더 추가 시마다 엔진 코드를 계속 수정해야 하는 부담이 있습니다.

## 🎯 SOAR 프로젝트를 위한 최종 아키텍처 제안

보안 관제 자동화(SOAR) 제품을 빠르게 완성하고 안정적으로 비즈니스를 운영하기 위한 가장 이상적인 조합은 `1번(Vector)`입니다.

* **이유**: C++이나 GoLang으로 수집기 자체를 만드는 것은 이미 시장에 나온 최고 수준의 오픈소스(Vector)를 하위 호환으로 재발명하는 일(Wheel 재발명)이 되기 쉽습니다.
* **권장 구조**: 바퀴를 직접 깎는(C++/Go 자체제작) 시간과 비용을 아껴 **`Vector`를 수집 엔진으로 채택**하고, 백엔드(NestJS)에서 Vector의 설정을 동적으로 제어(Hot Reload)하는 **'관리 기능 및 자동화 비즈니스 로직' 개발에 역량을 집중**하는 것이 시장 출시 속도와 제품 안정성 측면에서 압도적으로 유리합니다.
