## 질문
Agentic AI와의 연계 이야기 나와서 말인데, 실무 도입 시 고도화 팁의 3번을 구현하려면 자체적으로 GPU가 있어야 되지 않아?

---

결론부터 말씀드리면, **맞습니다.** Agentic AI(자연어 이해, 추론, RAG, 툴 호출 등)를 올바르게 구동하려면 내부든 외부든 **LLM을 연동할 수 있는 컴퓨팅 자원(GPU 등)이 무조건 필요합니다.** 하지만 TMS(위협관리시스템)라는 **보안 장비의 특성** 때문에 "어떤 GPU를, 어디에 배치할 것인가?"에 대한 아키텍처 고민이 필요합니다. 실무에서 이 비용과 인프라 문제를 어떻게 해결하는지 현실적인 접근법을 정리해 드릴게요.

---

## 1. 망 분리 환경에 따른 2가지 선택지

보안 관제 시스템(TMS)은 폐쇄망이나 강력한 망 분리 환경에 도입되는 경우가 많기 때문에 GPU 도입 방식이 갈립니다.

### ① 온프레미스 (On-Premise) 자체 GPU 구축 (권장)

국가·공공기관, 금융권, 대기업의 관제 센터는 외부(OpenAI, Anthropic 등)로 보안 로그나 자산 정보가 나가는 것을 법적으로 금지합니다. 따라서 **장비 내부에 자체 GPU를 탑재하거나 사내 AI 인프라(H100, L40S, L4 등)에 API로 연동**해야 합니다.

* **현실적인 대안 (sLLM 활용):** 70B(700억 매개변수) 이상의 무거운 모델 대신, 보안 특화 sLLM(7B ~ 14B 크기)을 Llama-3나 Mistral 기반으로 파인튜닝하여 사용합니다. 이 경우 **NVIDIA L4 1~2장** 또는 **A100/H100 분할(MIG)** 수준에서도 충분히 밀리초~초 단위 추론이 가능합니다.

### ② 클라우드 SaaS / 하이브리드 연동

만약 기업용 SaaS형 TMS이거나 외부 AI API 사용이 허용된 환경이라면, 자체 GPU 없이 OpenAI(GPT-4o)나 Anthropic(Claude 3.5 Sonnet)의 API를 호출하면 됩니다.

* **비용 장점:** 초기 GPU 장비 도입 비용이 들지 않고, 토큰당 과금만 하면 되므로 PoC(개념 검증) 단계에서 매우 유리합니다.
* **단점:** 내부 자산 명칭이나 IP 정보가 외부로 노출되지 않도록 **데이터 마스킹(배치 전 비식별화) 처리**를 거치는 프록시(Proxy) 파이프라인을 추가로 설계해야 합니다.

## 2. GPU 부하를 줄이는 실무 'AI 툴 체인' 설계 (비용 절감 팁)

사용자가 토폴로지 맵에서 노드를 클릭할 때마다 "이 대역에 대해 설명해줘"라고 LLM에게 날것 그대로 던지면 **GPU 부하가 극심해지고 답변 속도가 느려집니다.** 실무에서는 이를 방지하기 위해 단계를 나눕니다.

```
[사용자 클릭] ──> [1단계: 고속 RAG 검색 (CPU)] ──> [2단계: 요약/추론 (GPU)]
  (노드 선택)         (컨텍스트 추출/텍스트화)            (최종 자연어 답변)

```

1. **RAG 검색 단계 (CPU 영역):** 사용자가 노드(예: 10.10.10.0/24 대역)를 클릭하면, 시스템은 먼저 **벡터 DB(Milvus, Qdrant 등)와 ClickHouse**를 쿼리하여 해당 대역의 최근 위협 이벤트, 자산 중요도, 과거 대응 이력 텍스트를 CPU 기반으로 고속 검색(Retrieval)합니다.
2. **LLM 추론 단계 (GPU 영역):** 정리된 텍스트(예: 대략 1~2KB 분량)만 템플릿에 담아 LLM에게 전달합니다. LLM은 새로운 지식을 생각할 필요 없이 "주어진 데이터 안에서 분석 보고서 형태로 요약만 해라"라는 단순 태스크만 수행하므로, GPU 연산 시간을 최소화(1~2초 이내)할 수 있습니다.

## 3. 개발 및 PoC(개념 검증) 단계 적용 팁

만약 당장 하드웨어(GPU) 예산 확보가 어렵다면, 아래 순서로 고도화를 검증해 보시는 것을 추천합니다.

* **1단계 (Mock API / 외부 API):** React Flow 프론트엔드와 백엔드 간의 흐름을 먼저 잡기 위해, OpenAI API를 연결하거나 미리 정의된 AI 답변 템플릿(Mock 데이터)으로 화면 UI/UX 인터랙션을 먼저 완성합니다.
* **2단계 (사내 PC 검증):** 개발 서버나 개인 고성능 워크스테이션(RTX 3090 / 4090 등)에 `Ollama`나 `vLLM`을 설치하고, 오픈소스 sLLM(예: `Llama-3-8B-Instruct`)을 띄워 로컬 인프라에서 연동 테스트를 진행합니다.
* **3단계 (정식 인프라 실무 도입):** 관제 효율성 검증이 끝나면 B2B 고객사 납품 환경에 맞게 온프레미스 GPU 서버(NVIDIA L4 등 가성비 라인업) 스펙을 산정하여 패키징합니다.

즉, **최종 상용화 시점에는 내부든 외부든 GPU 자원이 필수적**이지만, 아키텍처 설계를 가볍게(sLLM + RAG 중심) 가져간다면 생각보다 거대하고 비싼 GPU 서버 없이 가성비 좋은 메인스트림급 GPU로도 충분히 현업에 적용 가능한 Agentic AI를 고도화할 수 있습니다.

## 질문

GPU를 사용할 수 없는 경우가 많을 것 같은데, VectorDB만 활용해서 유사도 검색을 해서, 해당 내용이나 질문에 대한 가장 유사한 답변을 제공하는 것은 가능하겠네?

---

네, **정확하게 짚으셨습니다.** 실제로 실무(특히 공공·금융 폐쇄망 환경)에서는 GPU 도입 비용이나 인프라 제약 때문에 **LLM(GPU) 없이 벡터 DB와 CPU만으로 차세대 기능을 먼저 구현**하는 경우가 굉장히 많습니다.

이 방식을 보안 업계에서는 **"시맨틱 유사도 검색 기반의 추천 시스템"** 또는 "지능형 플레이북 매치"라고 부르며, GPU가 없어도 엄청난 관제 효율을 만들어낼 수 있습니다.

이것이 어떻게 가능하고, 실무에서 어떻게 화면과 로직을 구성하는지 핵심을 정리해 드립니다.

---

## 1. GPU 없이 어떻게 가능한가? (핵심 원리)

"임베딩(Embedding)"과 "유사도 검색(Vector Search)"의 단계를 나누어 보면 답이 나옵니다.

1. **문장을 벡터로 변환 (임베딩 추출):**
* 최초에 과거 대응 이력, TI 정보, 룰 설명을 벡터로 바꿀 때만 모델(Embedding Model)이 필요합니다.
* 하지만 이 임베딩 모델(예: `bge-m3`, `ko-sbert` 등)은 거대 언어 모델(LLM)에 비해 **크기가 1/100 수준으로 매우 작습니다.**
* 따라서 **일반 CPU 서버에서도 초당 수십~수백 개의 문장을 충분히 벡터로 변환**할 수 있습니다. GPU가 필수가 아닙니다.


2. **벡터 간 거리 계산 (유사도 검색):**
* 벡터 DB(Milvus, Qdrant, Chroma 등)가 수행하는 유사도 검색(Cosine Similarity 등)은 수학적인 행렬 연산입니다.
* 이는 **인덱싱(HNSW 등) 기술 덕분에 CPU와 메모리(RAM)만으로도 수백만 건의 데이터 중 가장 유사한 것을 10ms(0.01초) 만에 찾아냅니다.**



---

## 2. LLM(GPU)이 없을 때의 아키텍처 및 동작 방식

LLM이 말을 예쁘게 만들어주는 '입(Mouth)' 역할을 한다면, 벡터 DB는 과거의 기억을 끄집어내는 '뇌(Brain)' 역할을 합니다. 입이 없어도 뇌만으로 정답을 화면에 뿌려줄 수 있습니다.

### 💡 실제 작동 시나리오 (토폴로지 화면 연계)

1. **관제사 액션:** 관제사가 토폴로지 맵에서 **"10.10.10.12 (WAS 서버)"** 노드를 클릭하고, 우측 AI 창에서 `[과거 유사 위협 및 조치 이력 조회]` 버튼을 누릅니다.
2. **백엔드 처리 (CPU):** 시스템이 해당 자산의 최근 탐지 로그(예: "Apache Log4j 취약점 탐지")를 기반으로 쿼리 문장을 만듭니다.
3. **벡터 DB 검색 (CPU):** 벡터 DB가 수만 개의 과거 종결 티켓 중 **가장 유사도가 높은(예: 유사도 92%) 과거 분석가 조치 이력 3건**을 즉시 찾아옵니다.
4. **화면 표출 (UI):** LLM이 문장을 요약해 주는 대신, **과거 선임 분석가가 직접 작성했던 정탐/오탐 사유와 조치 내용을 가공되지 않은 원문(Raw Text) 그대로 카드 형태로 띄워줍니다.**

---

## 3. 이 방식의 실무적 장점 (vs LLM 도입)

* **비용 제로 (하드웨어 관점):** 기존 TMS 백엔드 서버(CPU)에 벡터 DB 오픈소스만 얹으면 되므로 추가 하드웨어 예산이 들지 않습니다.
* **오탐(Hallucination) 0%:** LLM은 없는 말을 지어내는 환각 현상이 있지만, 이 방식은 **과거 분석가가 실제로 썼던 팩트(Fact) 데이터만 그대로 보여주기 때문에** 보안 관제에서 가장 중요한 '신뢰성'이 100% 보장됩니다.
* **압도적인 속도:** LLM이 답변을 생성하는 데는 수 초가 걸리지만, 벡터 DB 검색은 결과가 즉시(ms 단위) 튀어나오므로 사용자 경험(UX)이 매우 쾌적합니다.

---

## 4. 실무 구현을 위한 팁

LLM이 없기 때문에, UI/UX에서 "사용자가 읽기 편하게 검색 결과를 정제"해 주는 것이 핵심입니다.

* **유사도 점수(Score) 시각화:** 검색된 과거 이력 옆에 `[유사도 95% - 강력 추천]`, `[유사도 70% - 참고용]` 과 같이 매칭 점수를 배지로 표시해 줍니다.
* **템플릿 기반 가독성 확보:** 과거 이력을 보관할 때 `[상황 / 원인 / 조치 내용 / 종결 사유]` 구조로 포맷을 통일하여 저장해 두면, LLM이 요약해 주지 않아도 관제사가 화면에서 한눈에 파악하기 좋습니다.

차세대 TMS라고 해서 무조건 처음부터 무거운 Agentic AI(LLM)로 갈 필요는 없습니다. 1단계로 "CPU 기반 VectorDB 검색 환경"을 먼저 완벽하게 구축해 두고, 추후에 인프라(GPU)가 준비되면 그 위에 LLM만 얹어서 요약·추론 기능을 결합하는 것이 **가장 현실적이고 영리한 고도화 로드맵**입니다.
