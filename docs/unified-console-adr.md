# Unified Console ADR (권한 기반 싱글 뷰 통합)

작성일: 2026-06-05  
상태: Proposed

## 1. 배경

현재 제품은 다음과 같이 두 개의 독립 GUI를 제공한다.

1. Master Console: 시스템 공급자(Admin) 전용
2. Tenant Console: 서비스 사용자(Tenant) 전용

멀티테넌트 모드가 비활성화된 환경에서는 운영자가 시스템 설정과 서비스 운영을 한 화면에서 처리하고 싶어 하는 요구가 반복적으로 발생한다.

## 2. 문제 정의

1. 멀티테넌트 OFF에서 사용자 기대는 단일 콘솔 UX에 가깝다.
2. 그러나 보안 경계(Admin/Tenant)와 기존 API/Guard 구조는 이미 분리되어 있다.
3. 단일 SPA로 완전 병합 시 인증 토큰, Guard, 감사로그 흐름 회귀 리스크가 크다.

## 3. 결정

완전 병합 대신 Unified Console Shell 방식을 채택한다.

1. GUI 뼈대는 하나의 Shell에서 제공한다.
2. 실제 기능 화면은 기존 Admin/Tenant 화면과 API를 재사용한다.
3. 메뉴는 Role 및 멀티테넌트 스위치 값에 따라 동적으로 노출한다.
4. 백엔드 최종 권한 검증은 기존 Guard 체계를 유지한다.

## 4. 목표 동작

### 4.1 멀티테넌트 ON

1. Master 로그인: 시스템 설정 메뉴만 노출
2. Tenant 로그인: 서비스 메뉴만 노출
3. Master의 Tenant 영역 접근은 명시적 대리 로그인(act-as)으로만 허용

### 4.2 멀티테넌트 OFF

1. 최고 관리자 로그인 시 시스템 설정 + 서비스 메뉴 동시 노출
2. 서비스 메뉴 접근은 system 테넌트 컨텍스트로 처리
3. act-as 발급/사용/종료 이벤트를 감사로그로 강제 기록

## 5. 아키텍처 원칙

1. 보안 경계 유지
- Admin API(/admin/*)와 Tenant API(/api/*)를 논리적으로 합치지 않는다.
- Guard 우회 대신 명시적 토큰 교환 흐름을 사용한다.

2. 권한 이중 검증
- 프론트 메뉴 숨김은 UX 제어일 뿐, 보안 제어는 아니다.
- 최종 허용은 백엔드 Guard와 Role 정책으로만 결정한다.

3. 감사로그 의무
- 대리 로그인 관련 CUD/세션성 이벤트를 포함해 actor, action, resource, tenant, timestamp를 기록한다.

4. 멀티테넌트 호환성
- ON/OFF 전환 시 동일 코드 경로에서 동작해야 하며, 분기 로직은 최소화한다.

## 6. 대안 비교

### 대안 A: Admin에 서비스 화면 중복 구현

장점
1. 단기적으로 보이는 단일 UX 구현 가능

단점
1. 화면/권한/감사로그 구현이 중복됨
2. 버그 및 회귀 지점이 2배 이상 증가

결론: 채택하지 않음

### 대안 B: Admin/Tenant 완전 단일 SPA 병합

장점
1. 물리적 앱 수 감소

단점
1. 토큰 모델(master/tenant) 통합 난이도 높음
2. Guard/라우트 회귀 리스크 큼
3. 기존 운영 배포 경로 분리 전략과 충돌 가능

결론: 당장 채택하지 않음

### 대안 C: Unified Shell + 기존 앱/도메인 재사용

장점
1. 사용자에게 단일 진입 UX 제공
2. 보안 경계와 기존 코드 자산 유지
3. 중복 개발 최소화

단점
1. 토큰 교환(act-as) 설계가 필요
2. 메뉴 권한 매핑을 정교하게 관리해야 함

결론: 채택

## 7. 구현 범위

### In Scope

1. Unified Shell 라우팅/네비게이션
2. 모드/역할 기반 메뉴 정책
3. Master -> system tenant act-as 토큰 교환 API
4. OFF 모드 단일 콘솔 UX

### Out of Scope

1. 기존 Admin/Tenant 기능의 대규모 재작성
2. 기존 Guard 삭제 또는 권한 단순화
3. 임시 하드코딩 기반 우회 접근

## 8. 리스크 및 대응

1. 리스크: 권한 누락으로 메뉴 또는 API 과다 노출
- 대응: 메뉴 권한 매트릭스와 백엔드 Guard 테스트를 동시 운영

2. 리스크: act-as 토큰 남용
- 대응: 짧은 TTL, 1회성 jti, 발급/폐기 감사로그, 관리자 권한 제한

3. 리스크: ON/OFF 전환 회귀
- 대응: ON/OFF 각각 로그인, 메뉴, 핵심 API 접근 스모크 테스트 자동화

## 9. 승인 기준

1. 멀티테넌트 OFF에서 최고관리자 1회 로그인 후 시스템/서비스 메뉴 모두 접근 가능
2. 멀티테넌트 ON에서 기존 Admin/Tenant 분리 UX와 권한 정책 유지
3. 대리 로그인 이벤트 감사로그 100% 기록
4. 보안 회귀 테스트 통과
